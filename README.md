GitHub Actions CI practice

Небольшой проект, в котором я разобрался с базовым CI через GitHub Actions и тем, как правильно строится процесс проверки кода перед merge.

Что здесь есть

- workflow на GitHub Actions (".github/workflows/main.yml")
- запуск pipeline по:
  - push в main
  - pull request в main
- простой test step (для проверки, что pipeline реально работает)

Как это работает

Сценарий такой:

1. Создаю feature-ветку
2. Делаю изменения и пушу её
3. Открываю pull request
4. Автоматически запускается pipeline (pull_request)
5. Если всё ок — можно мержить
6. После merge снова запускается pipeline (push)

Зачем это делал

Хотел руками пройти базовый DevOps flow:

- понять разницу между push и pull_request
- увидеть, как реально запускаются pipeline'ы
- настроить защиту main ветки

Branch protection

Для ветки main включено:

- обязательное прохождение pipeline перед merge

То есть нельзя залить код в main, если тесты не прошли.

Что понял

- push — это уже после того, как код попал в ветку
- pull_request — это проверка до merge (самое важное)
- нормальный процесс всегда строится вокруг PR, а не просто push
- без branch protection CI почти не имеет смысла

Дальше хочу добавить

- реальные тесты (не echo)
- билд (например, Node.js или Python)
- деплой в AWS (через GitHub Actions)

---

Если кратко: это не “учебный пример ради примера”, а попытка руками собрать базовый CI процесс, как это делается в реальной работе.
